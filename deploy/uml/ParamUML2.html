Subsystem In UML models, subsystems are a type of stereotyped component that represent independent, behavioral units in a system. Subsystems are used in class, component, and use-case diagrams to represent large-scale components in the system that you are modeling



































Relationship in Use case diagrams There are four relationship can be used in Use Case diagram

1. Association ( between actor and use case elements)
2. Generalization ( between actors or between use cases)
3. Extend ( between use cases)
4. Include ( between use cases )
















he include relationship supports the reuse of functionality in a use-case model.

You can add include relationships to your model to show the following situations:

* The behavior of the inclusion use case is common to two or more use cases.
* The result of the behavior that the inclusion use case specifies, not the behavior itself, is important to the base use case.

Sheet 4: activity

Activity diagram history Before UML 2.0, activity diagrams were part of state machines and were referred to as activity graphs. In UML 2.0 the activity diagram is a distinct diagram, separate from a state machine, that focuses on the tasks that a system must perform.








































Use of activity diagram You can use activities to do the following things:

* Describe methods that correspond to the operations on classes
* Create organizational models of business processes where events can originate from inside or outside the system
* Create information system models that specify the system-level processes that occur
* describe the use-case functionality.

Sheet 10: Component Dia

Component Diagram used to break the software into manageable subsystem of a large system


















A Component is Replaceable, Executable piece of a large system whose implementation details are hidden


















The functionality of a component is specified by a set of Provided Interfaces ( Think as EJB's component interface )


















Required interface are, interaces used to get interact with another's component's provided interface.


















So provided interface are must for every component. But Required interface are option only need when to get interact with other component

Sheet 12: Profile, Sterio and Tag
Profile Profile is higher level of abstraction of system than UML.

It is defined by Sterio Type like we define class
Sterio Type will have Tagged value like attribute in Class diagram
Constraint can be applied for Tagged value .

But Sterio type, Tagged value and Constraints are not only used with profile defintion. It can be used also with any UML element
The advantage of profile is, whenever need the same profile for more than UML Class elment , i can reuse that profile with more than one class diagram
Sterio Type it give role differentiation for UML elements
Element can have more than one sterio type
Tagged Value Tag values are simply information for a particulare sterio type
This is name=value pair defined with in { }

Mostly Tagged value used when Sterio type used in a element to give role specific information
So, Tag values are simply information about sterio type rather than about element
This is high level abstraction. So, these are may not used in Code generation
This is used mainly for modeling purpose. But it might also used for code generation even not mandatory
Constraint This is used for tag values when element want interact with its neighbors
