11: SQL _ Tuning





Clause Benefit How To Benefit
SUB QUERIES Readability Use Sub quires to improve readability it is easy to debug, it is alternative for join most of time
SUB QUERIES Performance Never use Correlation sub query

WHERE Performance Don't use of literal characters comparison in where class, So get corresponding ID or any numeric value and use in where class Since, all the character are converted into numeric value before to compare
WHERE Readability Use BETWEEN, IN instead many AND, OR condition easy to maintain the query. No performance impact / increase
WHERE Readability Use = operator instead of LIKE if we are not using any
Of wild character ( %, _ )		 easy to maintain the query. No performance impact / increase
WHERE Performance use unique key, primary key, indexed column, try to avoid other column in WHERE clause if possible it will improve the speed
WHERE Performance use IS NULL, will scan entire table if that is key or indexed column so avoid to use of IS NULL in WHERE clause
SELECT, ORDER BY Readability Use alias name for any expression in select & Order by clause easy to maintain the query. No performance impact / increase
Design Readability It is always good to have a Active_status_flag column in every table both development time as well as production, even it is not necessary for UI

TABLE Performance use appropriate exact data type for column definition to save memory space

TABLE Performance never use LONG data type to column since only one column can be LONG, in a table, plz avoid it
TABLE Performance never drop column one by one. Instead of set UN USED dropping a column is costly. So, just mark it as UN USED, and at leisure time just remove all unused column at single pass
VIEW Readability create view, if the select query is too complex. it will easy to other developers to select data from view
JOIN Performance While join always use to table prefix to improve the performance even though there is no ambiguity among columns Oracle save the execution time, by avoiding resolve the column reference
JOIN Readability If more than one 2 tables to be joined, go for Join.. Using() we must know the common column name
JOIN Readability If join only 2 tables, go for Natural Join we must know the common column name
JOIN Readability If join, arbitrary column name, express go for join.. ON()

JOIN Readability Use BETWEEN in JOIN.. ON(), to avoid AND, >=,<=

JOIN
try to avoid usage of natural join, since when matching column not found, it will cross join

PL/SQL Performance use appropriate exact data type for Variable definition to save memory space

Set Performance use UNION ALL if you are sure, no duplicate, instead of UNION. UNION will use little more resource than UNION ALL
TYPE Readability Use explicit type casting it easy to read and debug
TYPE Performance Use explicit type casting it save database instance, doing some task for implicit type casting
INSERT Readability use column name in Insert query instead of depends on positional it is easy to debug and understanding, Even if the table column altered, it wont create any bug
INSERT Readability don't use USER for value of user name or id column. it is not good for web application, since different user use same user name to login DB




















INDEX Performance Keep the indexes minimal. index help only for data retrieval, But reduce the performance for DML
