Sheet 3: Prior to Hibernate
Database - relational technolgy



Industry has invested a lot for efficent ralational database system.

Relational technology is good interms of data independency, protection, integrity,sharing

All Database accessed only by SQL only. But SQL is not implemented any of relational technology concept

SQL already solved by relational technology and has extension by PL/SQL procedure

Usally every database has its own Dialect.




Database - SQL

Data Definition Language ( DDL ) - CREATE / ALTER statement

Data Manipulation Language ( DML ) - INSERT / UPDATE / DELETE / SELECT ( join / restriction / projection / reporting like group, order,aggregate )




Paradigm Mismatch

Object modelling Vs Relational Modeling ( usually both model is must of a non-trival application development )

Object oriented progamming ( fully use of OOPs ) Vs SQL

Java object is best for business logic, since it use OOPS concept & Design pattern Vs it cannot be run inside database like pl/ SQL procedure

for data persistence Relational is best Vs serialization java objects

Use SQL Procedure when modify large data (i.e) close to data Vs Bring data to application layer when regular use of business (i.e) close to business logic

Relational Database does not support User Defined Type (UDT) to specify column of type. (i.e) a java class cant specify as type of a column in table

Relational Database does not support concept like Super table and sub tables Vs Java

Relational Database does not support Polymorphic association or polymorphic query result Vs Java

Identity of a record depends on primary key Vs Identity of java objects depends on memory reference and equals().

Java object association is by object reference Vs Tables are associated with Foreign key

Java reference is directional Vs foreign key Is not directonal by default

tables arbitarly joined by tables join and projection even without foreign key association in Select query. Vs Association cant without reference among objects

one-to-one/ one-to-many/ many-to-many all are supported in Java by object reference Vs Foreign key support only one-to-one / one -to-many. For many-to-many need LINK table

some time Domain model and Object model cannot be the same. This mismatch need a couple of cost in wring SQL and JDBC


Java

Java object can be serialized and recreated even single or networks of interconnected object.

But a java application have mixed of both serializable as well as Transient object.

so we need a subsystem( database ) to manage the persistence data






Datbase ( subsystem) Provides

it represent data in structured manner

it provide data integrity & Concurrency by constraint rules

it provide data sharing among different user / application

it provide data manipulation, sorting, grouping(aggregation)

it provide data level security




Layered architecture

each layer aware only below of its.

each layer communicate only with below of it

Sheet 5: Extension point
1 Id Generation we can plug our own implementation to generate ID generation, if we need like to append any character with number
2 ConnectionProvider we can plug our own implementation to SessionFactory if we need to supply our own DB connection hibernate configuration
3 TransactionFactory we can plug our own implementation of TransactionFactory
4 CacheProvider we can plug our own cache provider implementation hibernate configuration
5 QueryCache we can plug our own query cache provider implementation hibernate configuration
6 exception converter we can plug our own SQLException converter
7 Dialect we can plug our own Dialect implementation hibernate configuration
8 EntityPersister we can plug our own entity persister for a class. For example to persit as flat file mapping
9 cutom Data type we can create new hibernate data type
10 property accessor we can plug our own property accessor mapping
11 session Context we can plug our own session context provider implementation hibernate configuration
